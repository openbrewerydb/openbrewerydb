# AGENTS.md — Open Brewery DB

## Project Overview

Open Brewery DB is an open-source dataset of breweries, cideries, brewpubs, and bottleshops. This repository contains the dataset (CSV, JSON, SQL) and TypeScript tooling to validate, split, combine, generate, and maintain the data. The dataset is served by a separate Laravel API at [openbrewerydb.org](https://www.openbrewerydb.org/).

```
  update-readme-stats.ts       # Updates the statistics section in README.md

  states.ts                    # US state name/abbreviation mappings
  templates/                   # Mustache templates for SQL generation

  dedupe-candidates.ts         # Candidate detection logic and hash-based identification
  dedupe-review.ts             # Interactive CLI for reviewing candidates
  dedupe-apply.ts              # Applies resolutions to data/ files
  analyze-data.ts              # Data quality analysis and reporting

  utils/                       # Shared utility modules

```

## Tech Stack

- **Runtime:** Node.js 22.x
- **Language:** TypeScript (ES2023, CommonJS modules)
- **Script runner:** `ts-eager` (runs TypeScript directly without pre-compilation)
- **Validation:** Zod 4.x
- **CSV parsing:** PapaParse 5.x
- **Other:** mustache (templates), puppeteer, pg-promise, slugify, uuid, date-fns, glob

## Common Workflows

### Full Maintenance (maintain)

Runs the complete pipeline: validate → combine → split → generate IDs → generate JSON/SQL → update README stats.

```bash
npm run workflow:maintain
```

### Validate Data

```bash
npm run validate
```

Validates all CSVs under `data/` and the combined `breweries.csv` against the Zod schema in `src/types.ts`. Reports per-brewery field-level errors.

### Add/Edit Brewery Data (the right way)

1. Edit the **individual CSV file** under `data/[country]/[state_province].csv`
2. Do **not** include an `id` field — it will be auto-generated
3. Do **not** edit `breweries.csv` directly (it's auto-generated)
4. Run `npm run validate` to check your changes
5. Run `npm run workflow:maintain` to regenerate all output files

### Regenerate Output Files

```bash
npm run csv:combine       # data/*.csv → breweries.csv
npm run csv:split         # breweries.csv → data/*.csv
npm run generate:ids      # Assign UUIDs to rows missing them
npm run generate:json     # breweries.csv → breweries.json
npm run generate:sql      # breweries.csv → breweries.sql
npm run generate:stats    # Print dataset statistics
npm run update:readme-stats  # Update README.md statistics section
```

### Data Quality Analysis

```bash
npm run analyze           # Analyze breweries.csv and generate data-quality-report.md
```

Analyzes the combined dataset for:

- Data completeness by field
- Potential duplicates (name, address, coordinates)
- Address format quality by country
- Brewery type distribution

The report is written to `data-quality-report.md` and automatically posted as a PR comment in CI.

### Address Normalization

```bash
npm run normalize:addresses  # Normalize address fields in individual CSV files
```

Normalizes address fields (address_1, address_2, address_3, city, state_province) across all CSV files under `data/`:

- US addresses: USPS street suffix abbreviations (Street → St, Avenue → Ave, etc.)
- US addresses: Secondary unit designators (Suite → Ste, Apartment → Apt, etc.)
- All addresses: Intelligent title casing (preserves German particles, Mc/Mac/O' patterns)
- Validation gate runs before any modifications

**Important:** After running `normalize:addresses`, run `npm run workflow:maintain` to regenerate `breweries.csv`, `breweries.json`, and `breweries.sql`.

### Fuzzy Deduplication

```bash
npm run dedupe           # Detect duplicate candidates using fuzzy matching
npm run dedupe:review    # Interactively review and confirm/reject candidates
npm run dedupe:apply     # Apply confirmed resolutions to data/ files
```

Detects potential duplicate breweries using fuzzy string matching (Jaro-Winkler similarity) and applies resolutions to clean the dataset.

**Workflow:**

1. `npm run dedupe` - Scans `breweries.csv` for duplicates based on:
   - Address normalization + Jaro-Winkler name similarity (≥85%)
   - Coordinate proximity (~50m) + name similarity
   - Quality scoring (phone, website, coordinates, name length)
2. `npm run dedupe:review` - Interactive CLI to:
   - Review each candidate pair with similarity score and confidence level
   - Confirm (remove duplicate), reject (keep both), or skip for later
   - Auto-approve high-confidence pairs (≥95% similarity)
   - Resolutions are stored in `dedupe-resolutions.json` with stable hash-based IDs
3. `npm run dedupe:apply` - Applies confirmed resolutions:
   - Filters out confirmed duplicates and updates `data/` files directly
   - Creates backup of `data/` before modifications with automatic rollback on error
   - Validates output record count before committing
   - Then run `npm run workflow:maintain` to regenerate `breweries.csv`, JSON, SQL, and README stats

**Idempotency:**

- Candidates use SHA-256 hash of (keepId, removeId) as stable identifiers
- Resolutions persist across re-runs even if candidates are regenerated
- Re-running `dedupe` after applying will show previously resolved pairs as "already resolved"
- Safe to re-run the entire workflow without double-counting or losing resolution state

**Artifacts (gitignored):**

- `dedupe-candidates.json` - Detected duplicate pairs
- `dedupe-resolutions.json` - User's resolution decisions
- `dedupe-report.md` - Markdown summary of candidates

## Code Conventions

- **TypeScript:** Strict mode enabled (`tsconfig.json`). Target ES2023, CommonJS modules.
- **Imports:** Use standard Node.js `require`/`import`. No path aliases beyond `baseUrl: "."`.
- **CSV handling:** Always use PapaParse with the options from `config.ts` (`papaParseOptions`). Empty strings are treated as `null`.
- **Validation:** All changes to `src/types.ts` (Zod schema) should be reflected in `src/config.ts` column headings and brewery types, and vice versa.
- **Formatting:** Prettier is configured (see `.prettierrc`).
- **Commit style:** Squash commits when possible (per `CONTRIBUTING.md`).

## Adding a New Country

1. Create a new directory under `data/` (e.g., `data/mexico/`).
2. Add a CSV file following the column order in `config.ts`.
3. Run `npm run validate` to ensure correctness.
4. Run `npm run workflow:maintain` to regenerate combined outputs.

## Updating the Zod Schema (Important)

If you add, remove, or change columns:

1. Update `src/config.ts` — `headers` array and/or `BREWERY_TYPES`
2. Update `src/types.ts` — `Brewery` Zod schema
3. Update any scripts that reference affected columns
4. Run `npm run validate` against existing data to check for breakage

## Key Gotchas

- `breweries.csv`, `breweries.json`, and `breweries.sql` are **auto-generated**. Never edit them directly.
- `postal_code` is always treated as a string via `z.coerce.string()`.
- The `id` field is skipped during validation of individual data files (IDs are added by the maintenance workflow).
- `dynamicTyping` in PapaParse explicitly sets `postal_code: false` to prevent numeric coercion.
- The `transform` in `papaParseOptions` trims whitespace and converts empty strings to `null`.
